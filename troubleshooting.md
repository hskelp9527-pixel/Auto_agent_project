# self-evolution-system 使用指南

## 🎯 如何正确唤醒

### ✅ 推荐的唤醒方式

```bash
# 最直接的方式
"用 self-evolution-system 验证这个项目"

# 或者使用触发关键词
"项目做完了，帮我验收"
"帮我测试一下这个项目"
"评估一下项目质量"
```

### ❌ 避免的唤醒方式

```bash
# 太模糊
"看看我的项目"
"项目写得怎么样"

# 不匹配 skill 功能
"帮我修复这个bug"  (skill 只检查，不修复)
"重构这段代码"      (skill 不是代码审查工具)
```

## 🔍 验证 Skill 是否正确执行

### 检查清单

当 skill 被触发后，应该看到以下执行流程：

#### ✅ 第0轮：ERROR_LOG.md 检查（必须）
```
📋 第0轮：错误历史检查

- 读取已记录的错误
```

## ⚠️ 常见问题与解决

### 问题1：Skill 只做静态分析，不做浏览器测试

**原因**：AI 可能误解了"静态95分≠真实功能95分"的规则

**解决方案**：
明确告诉 AI：
```
"必须启动浏览器实际测试，不能只看代码。静态分析95分后必须继续浏览器验证。"
```

### 问题2：Skill 在75分就询问"是否启动"

**原因**：AI 违反了"禁止中间分数询问"的规则

**解决方案**：
明确告诉 AI：
```
"不要在75分或任何中间分数询问我。继续验证直到95分真实功能。"
```

### 问题3：Skill 跳过第0轮ERROR_LOG检查

**原因**：AI 没有按照从第0轮开始的顺序执行

**解决方案**：
明确告诉 AI：
```
"必须从第0轮ERROR_LOG.md检查开始。"
```

## 🎯 Skill 的职责范围

### ✅ Skill 会做的：
- 检查项目完整性
- 验证功能可用性
- 测试设计系统质量
- 评估代码质量
- 浏览器功能验证
- 给出评分和改进建议

### ❌ Skill 不会做的：
- 自动修复问题
- 重构代码
- 添加新功能
- 性能优化
- 部署项目

---

## 💡 最佳实践

1. **项目完成后再使用**：在所有功能开发完成后使用
2. **明确期望**：告诉 AI 你想要什么样的验证深度
3. **提供上下文**：如果有 ERROR_LOG.md，确保它存在
4. **准备修复问题**：skill 只检查，你需要根据建议修复问题
5. **迭代使用**：修复问题后再次使用 skill 验证

## 🔧 如果 Skill 还是不按预期工作

### 方案1：手动指定执行步骤
```bash
"请按以下步骤验证项目：
1. 读取 ERROR_LOG.md（第0轮）
2. 检查内容完整性（第1轮）
3. 检查设计系统（第2轮）
4. 检查代码质量（第3轮）
5. 启动浏览器测试（第4轮）
6. 给出总分和建议"
```

### 方案2：分步执行
```bash
# 先做第0轮
"先检查ERROR_LOG.md"

# 然后做第1-3轮
"现在检查内容、设计、代码质量"

# 最后做第4轮
"现在启动浏览器验证功能"
```

### 方案3：使用 Skill 工具直接调用
```bash
/skill use self-evolution-system
```
