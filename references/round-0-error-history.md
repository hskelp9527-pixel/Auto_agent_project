# 第0轮：错误历史检查 (error-history-check)

**优先级**: ⚠️ **最高优先级 - 必须首先执行**

## 目标

避免重复已知的错误，确保ERROR_LOG.md成为"活文档"而不是"死档案"。

## 为什么这是第0轮？

- **不重复错误是最基本的要求**
- ERROR_LOG.md记录的错误必须被读取和应用
- 如果历史错误未修复，先修复再继续
- 这是"从错误中学习"的具体体现

## 执行步骤

### 步骤1：检查ERROR_LOG.md是否存在

```bash
# 检查ERROR_LOG.md是否存在
if [ -f "ERROR_LOG.md" ]; then
    echo "✅ 发现ERROR_LOG.md，开始分析..."
else
    echo "⚠️ 未发现ERROR_LOG.md，创建新文件"
    touch ERROR_LOG.md
    # 初始化ERROR_LOG.md模板
    cat > ERROR_LOG.md << 'EOF'
# 错误日志 - [项目名称]

> **重要**: 本文件记录所有遇到的错误。不要重复同样的错误两次。

---

## 🔴 错误记录

### 错误 #001
**时间**: [时间戳]
**错误**: [错误描述]
**原因**: [根本原因]
**解决方案**: [如何修复]
**预防措施**: [如何预防]
**状态**: [已解决/未解决]

---

## 📊 错误统计

| 类别 | 数量 | 状态 |
|------|------|------|
| 配置错误 | 0 | - |
| 代码规范 | 0 | - |
| 逻辑错误 | 0 | - |
| 性能问题 | 0 | - |
| **总计** | **0** | **0 已解决** |

---

**记住: 错误是最好的老师，但不要重复同样的错误！** 🚫🔄
EOF
fi
```

### 步骤2：分析已记录的错误

```bash
# 读取所有已记录的错误
grep -A 10 "### 错误 #" ERROR_LOG.md
```

**检查内容**：
- 错误总数
- 已解决数量
- 未解决数量
- 错误类型分布

### 步骤3：检查是否已修复

对于每个已记录的错误：

**✅ 状态为"已解决"的错误**：
- 确认是否真的修复（通过测试验证）
- 检查修复方案是否正确执行
- 验证预防措施是否落实

**❌ 状态为"未解决"的错误**：
- **必须先修复这些错误**
- 不能继续第1轮
- 修复后更新ERROR_LOG.md状态

### 步骤4：检查是否有类似问题

根据已记录错误的"预防措施"检查当前项目：

```bash
# 示例：如果记录了CSS颜色对比度问题
# 检查当前CSS是否有类似问题
grep -r "color.*rgba(255" src/ --include="*.css"
grep -r "background-color.*#ffffff" src/ --include="*.css"
```

**其他常见检查**：
- 配置错误：检查相关配置文件
- 代码规范：运行linter检查
- 逻辑错误：检查相关代码逻辑
- 性能问题：运行性能分析

**发现类似问题时**：
- **立即修复**
- 记录到ERROR_LOG.md
- 不能继续第1轮

### 步骤5：更新ERROR_LOG.md（如需要）

如果发现新的类似错误：
1. 记录到ERROR_LOG.md
2. 更新错误统计
3. 添加预防措施

## 通过标准

- ✅ ERROR_LOG.md中所有"未解决"的错误已修复
- ✅ ERROR_LOG.md中所有"已解决"的错误确认修复（通过测试）
- ✅ 当前项目没有ERROR_LOG.md中记录的类似问题

## 不通过标准

- ❌ 有未解决的错误 → **立即修复，不能继续第1轮**
- ❌ 发现类似问题 → **立即修复，不能继续第1轮**

## 输出要求

每次第0轮检查后，必须输出：

```bash
📋 第0轮：错误历史检查
- ERROR_LOG.md中的错误总数：X
- 已解决：Y，未解决：Z
- 发现的类似问题：[列出]
- 修复的问题：[列出]
- 状态：✅ 通过 / ❌ 不通过
```

## 重要说明

- ⚠️ **第0轮不通过，绝对不能进入第1轮**
- ⚠️ **每次迭代都要从第0轮开始**
- ⚠️ **ERROR_LOG.md是"活文档"，必须被读取和使用**
- ❌ **禁止跳过第0轮直接进入第1轮**
- ❌ **禁止只记录不读取ERROR_LOG.md**
