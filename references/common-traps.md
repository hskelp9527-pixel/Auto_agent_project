# 常见陷阱

**目的**: 避免最常见的错误，确保系统正确运行。

---

## 🔴 最严重的错误

### 1. 用上下文传递信息 ❌

**错误行为**:
```
AI: "我刚才完成了任务#001，现在开始任务#002"
```

**为什么错误**:
- 依赖上下文，换个 AI 就无法继续
- 没有文件记录，无法追溯

**正确做法**:
```
AI: 读取 task.md → 任务#001
AI: 更新 progress.md → 任务#001完成
AI: 读取 task.md → 任务#002
```

**核心原则**: **所有信息必须写在文件里！**

---

### 2. 保留减分的代码 ❌

**错误行为**:
```
AI: "评估结果是48分，减了4分。我先保留这个版本，下次再优化"
```

**为什么错误**:
- 违反迭代原则
- 减分的代码应该直接重写

**正确做法**:
```
AI: 读取 evaluation.md → 48分（-4）
AI: 直接删除代码
AI: 重新开发任务#002
AI: 再次请求评估
```

**核心原则**: **减分直接重写，不留垃圾代码！**

---

### 3. 跳过 ERROR_LOG.md 检查 ❌

**错误行为**:
```
AI: "开始任务#015开发"
（直接开始写代码，没有读取 ERROR_LOG.md）
```

**为什么错误**:
- 可能重复已知的错误
- 违反第0轮最高优先级规则

**正确做法**:
```
AI: 读取 ERROR_LOG.md
AI: 发现有CSS颜色对比度问题
AI: 开发时避免这个问题
AI: 开始任务#015
```

**核心原则**: **每次开发前必须读 ERROR_LOG.md！**

---

### 4. 在中间分数询问用户 ❌

**错误行为**:
```
AI: "当前分数75分，您想现在启动看看效果，还是继续优化到95分？"
```

**为什么错误**:
- 违反自主迭代原则
- 打断开发流程

**正确做法**:
```
AI: 当前分数75分
AI: 读取 task.md → 下一个任务#020
AI: 继续开发
```

**核心原则**: **持续迭代到95分，中间不询问！**

## ⚠️ 常见错误

### 5. 任务拆解太粗 ❌

**错误行为**:
```markdown
### 任务 #001 - 开发整个项目
**描述**: 完成所有功能
**预计分数**: 100分
```

**为什么错误**:
- 无法迭代
- 无法评分
- 无法重写

**正确做法**:
```markdown
### 任务 #001 - 初始化项目
**描述**: npm create vite@latest
**预计分数**: +5分

### 任务 #002 - 添加基础布局
**描述**: 创建 Layout 组件
**预计分数**: +10分
```

---

### 6. 没有独立评估 ❌

**错误行为**:
```
AI: "我觉得这个任务做得不错，应该有80分"
```

**为什么错误**:
- 开发和评估是同一个 AI
- 缺乏客观性
- 不符合系统设计

**正确做法**:
```
开发 AI: 完成任务#015
开发 AI: 调用评估 agent
评估 agent: 读取项目文件
评估 agent: 打分并写入 evaluation.md
开发 AI: 读取 evaluation.md
```

---

### 7. 文件格式不统一 ❌

**错误行为**:
```markdown
# task.md（格式1）
## 任务1
- 描述：xxx
- 分数：5

# task.md（格式2）
### Task #002
**Description**: xxx
**Score**: 10
```

**为什么错误**:
- AI 读取困难
- 容易解析错误

**正确做法**:
统一使用标准格式（见 SKILL.md 中的文件结构部分）

---

### 8. 没有记录分数变化 ❌

**错误行为**:
```markdown
## 任务 #001
**状态**: 已完成
```

**为什么错误**:
- 无法追溯分数变化
- 无法判断是否加分

**正确做法**:
```markdown
## 任务 #001
**状态**: 已完成
**分数变化**: 0 → 5 (+5)
```

---

### 9. 忽略依赖关系 ❌

**错误行为**:
```markdown
### 任务 #003 - 添加产品详情页
**依赖**: 无

### 任务 #005 - 在产品列表添加详情页链接
**依赖**: 无
```

**为什么错误**:
- 任务#005依赖任务#003
- 顺序错了会导致问题

**正确做法**:
```markdown
### 任务 #003 - 添加产品详情页
**依赖**: 无

### 任务 #005 - 在产品列表添加详情页链接
**依赖**: #003
```

---

### 10. 没有错误预防措施 ❌

**错误行为**:
```markdown
### 错误 #001
**错误**: CSS颜色对比度问题
**状态**: 已解决
```

**为什么错误**:
- 没有预防措施
- 可能再次犯同样错误

**正确做法**:
```markdown
### 错误 #001
**错误**: CSS颜色对比度问题
**预防措施**: 检查 color-scheme 设置，确保亮色主题用深色文字
**状态**: 已解决
```

## ✅ 最佳实践

### 1. 文件驱动开发 ✅

**原则**: 所有信息都在文件里

**实践**:
```bash
# 开发前
读取 task.md → 获取任务
读取 progress.md → 了解进度
读取 ERROR_LOG.md → 避免错误

# 开发后
更新 task.md → 标记完成
更新 progress.md → 记录分数
请求评估 → 生成 evaluation.md
```

---

### 2. 小步快跑 ✅

**原则**: 每个任务小而独立

**实践**:
- 每个任务 +5-15分
- 任务独立可验证
- 容易重写

---

### 3. 频繁评估 ✅

**原则**: 每个任务完成后立即评估

**实践**:
```
完成任务 → 请求评估 → 读取 evaluation.md
加分 → 继续
减分 → 重写 → 再评估
```

---

### 4. 记录一切 ✅

**原则**: 所有变化都要记录

**实践**:
- progress.md 记录每次分数变化
- evaluation.md 记录每次评估详情
- ERROR_LOG.md 记录每个错误和预防措施

---

### 5. 信任评估 ✅

**原则**: 评估 agent 的判断是最终标准

**实践**:
- 不质疑评估结果
- 减分就重写
- 加分就继续

---

## 🎯 检查清单

### 开发任务前检查
- [ ] 读取 task.md
- [ ] 读取 progress.md
- [ ] 读取 ERROR_LOG.md
- [ ] 确认任务依赖已满足

### 开发任务后检查
- [ ] 代码已提交
- [ ] 更新 task.md（状态）
- [ ] 请求评估
- [ ] 读取 evaluation.md

### 评估后检查
- [ ] 如果加分：更新 progress.md
- [ ] 如果减分：直接重写
- [ ] 如果不变：更新 progress.md

### 交付前检查
- [ ] 分数 ≥ 95
- [ ] 所有 P0 任务完成
- [ ] ERROR_LOG.md 所有错误已解决
- [ ] 无404、无按钮失效、无占位符

## 📝 快速参考

### 文件必须包含的信息

**task.md**:
- 任务ID、标题、描述
- 预计分数
- 依赖关系
- 状态

**progress.md**:
- 当前分数
- 当前任务
- 已完成任务列表
- 每个任务的分数变化

**evaluation.md**:
- 当前分数
- 上次分数
- 分数变化
- 4个维度详细评分
- 修改意见
- 下一步建议

**ERROR_LOG.md**:
- 错误描述
- 根本原因
- 解决方案
- **预防措施**
- 状态
