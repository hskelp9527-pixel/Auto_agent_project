# Web应用上线前自我迭代系统 (Self-Evolution System for Web App QA)

基于 [autoresearch](https://github.com/karpathy/autoresearch) 理念，为Web应用开发项目设计的上线前质量保证框架。

## 🎯 核心理念

**源自 autoresearch (Andrej Karpathy)**：
> "Research is now entirely the domain of autonomous swarms of AI agents"

**应用到 Web 应用 QA**：
> "Quality Assurance is now entirely the domain of autonomous AI agents before user sees it"

## 📖 什么是 autoresearch？

[autoresearch](https://github.com/karpathy/autoresearch) 是 Andrej Karpathy 创建的项目，让 AI 代理自主进行机器学习研究：
- **自主迭代**：AI 修改代码 → 训练 5 分钟 → 评估结果 → 保留或丢弃
- **持续改进**：无人值守的持续优化，醒来后看到更好的模型
- **实验追踪**：完整的 git 历史和结果记录

## 🔄 我们的改造

autoresearch 的核心思想（AI 自主迭代）不仅适用于 ML 研究，也适用于 **Web 应用开发**：

### autoresearch 原版
- **领域**：机器学习研究
- **目标**：优化模型性能（降低 val_bpb）
- **方法**：修改 `train.py` → 训练 5 分钟 → 评估
- **输出**：更好的模型

### 我们的改造（Web 应用 QA）
- **领域**：Web 应用开发
- **目标**：确保上线前质量（零 bug、零 404、功能完整）
- **方法**：多轮迭代验证（内容完整性 → 设计系统 → 代码质量 → 浏览器功能）
- **输出**：可立即交付的高质量应用

## 🎯 增加的功能

相比 autoresearch，我们增加了：

### 1. 需求确认阶段
- **第1轮迭代**：内容完整性检查（权重 60%）
  - 页面完整性（无占位符）
  - 表单完整性（字段完整）
  - 示例数据（足够数量）
  - CRUD 功能（按钮可用）
  - **路由存在性**（零 404）

### 2. 项目上线前测试
- **第2轮迭代**：设计系统质量（权重 20%）
- **第3轮迭代**：代码质量检查（权重 10%）
- **第4轮迭代**：浏览器功能验证（权重 10%）

### 3. 零容忍规则
- ❌ 404 错误零容忍
- ❌ 按钮无反应零容忍
- ❌ 占位符内容零容忍

## 📦 包含的 Skills

### 1. self-evolution-system (核心)
**自我迭代系统** - 基于 autoresearch 理念的四轮迭代验证流程

**四轮迭代**：
- 第1轮：内容完整性检查（60分）⭐ 核心
- 第2轮：设计系统质量（20分）
- 第3轮：代码质量检查（10分）
- 第4轮：浏览器功能验证（10分）

**何时使用**：
- 完成任何 Web 应用开发后
- 用户要求"启动项目"、"部署"之前
- 任何即将呈现工作成果给用户的时刻

### 2. ready-to-use-verification
**开箱即用验证** - 浏览器功能测试

**核心流程**：
1. 启动开发服务器
2. 使用**有头浏览器**打开（必须能看到浏览器窗口）
3. 验证首页和所有导航链接
4. 测试核心 CRUD 功能
5. 检查浏览器控制台无错误

**铁律**：
```
NO "READY" CLAIMS WITHOUT HEADFUL BROWSER VERIFICATION
```

### 3. verification-before-completion
**完成前验证** - 通用验证原则

**核心原则**：Evidence before claims, always（证据优先于声明）

**验证流程**：
1. IDENTIFY: 什么命令能证明这个声明？
2. RUN: 执行完整的命令
3. READ: 读取完整输出
4. VERIFY: 输出是否确认声明？
5. ONLY THEN: 只有这样才能做出声明

### 4. webapp-testing
**Web 应用测试** - 自动化测试工具包

**主要功能**：
- Playwright 脚本编写
- 服务器生命周期管理
- DOM 检查和元素发现
- 控制台日志捕获


## 🔧 实际案例

### 案例：订单管理系统

**基于项目经验发现并修复的问题**：
1. ❌ 客户端组件错误使用 async
2. ❌ 重复的 aria-label 属性
3. ❌ aria-level 类型错误
4. ❌ 重复属性和混乱的 className
5. ❌ StoreState 未导出、persist 配置错误
6. ❌ **CRUD 按钮不可用**（只有 UI，无功能）
7. ❌ **编辑页路由不存在**（出现 404 错误）
8. ❌ **占位符内容**（"订单表单"、"库存列表"）

**系统改进**：
- ✅ 新增"功能可用性检查"（5分）
- ✅ 新增"路由存在性检查"（零容忍）
- ✅ 所有检查项改为通用表述
- ✅ 适用于任何 Web 应用项目

**结果**：
- 从 29.5/60 → 60/60（内容完整性）
- 总分：95/100
- 用户看到的是完全可用的系统

## 🆚 与 autoresearch 的对比

| 维度 | autoresearch (原版) | 自我迭代系统（我们的改造） |
|------|-------------------|------------------------|
| **核心理念** | AI 自主迭代研究 | AI 自主迭代 QA |
| **应用领域** | 机器学习研究 | Web 应用开发 |
| **迭代对象** | `train.py` (模型代码) | 整个 Web 应用 |
| **评估标准** | val_bpb (验证损失) | 多维度评分（100分制） |
| **时间预算** | 固定 5 分钟 | 不限制（直到达到标准） |
| **输出结果** | 更好的模型 | 可交付的应用 |
| **人类角色** | 设置 `program.md` | 设置项目标准 |
| **验证方式** | 自动训练 | 4轮迭代验证 |
| **零容忍** | 无 | 404、按钮失效、占位符 |

## ⚡ 核心原则

1. **证据优先**: 不相信"应该可以"，只相信实际测试结果
2. **用户视角**: 从用户角度测试，不是开发者角度
3. **完整性**: 测试所有声称可用的功能
4. **诚实**: 发现问题如实报告，不隐瞒
5. **迭代**: 发现问题 → 修复 → 重新验证 → 呈现

## 🚫 禁止行为

❌ **不要**:
- 只检查代码就说"ready"
- 只看首页就说"完成"
- 用无头浏览器就说"测试过了"
- 跳过任何验证阶段
- 隐瞒发现的问题

✅ **要**:
- 完整执行四个轮次
- 使用有头浏览器
- 测试所有功能
- 保留截图证据
- 发现问题立即修复

## 💡 时间估算

| 项目规模 | 第1轮 | 第2轮 | 第3轮 | 第4轮 | 总计 |
|---------|------|------|------|------|------|
| 小型 | 5-10分钟 | 3-5分钟 | 2-3分钟 | 5-10分钟 | 15-28分钟 |
| 中型 | 15-20分钟 | 5-10分钟 | 5分钟 | 15-20分钟 | 40-55分钟 |
| 大型 | 30-40分钟 | 10-15分钟 | 10分钟 | 30-40分钟 | 80-105分钟 |

**原则**: 宁可多花 1 小时验证，不要让用户发现 1 个问题。
