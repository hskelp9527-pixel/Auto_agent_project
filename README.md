# 自我迭代系统 Skills (Self-Evolution System)

这是一个完整的自我迭代验证系统，确保在呈现给用户之前，所有 Web 应用都经过严格的质量验证。

## 📦 包含的 Skills

### 1. self-evolution-system (核心)
**自我迭代系统** - 整合三阶段验证流程的主 skill

- **阶段1**: 代码级验证 (verification-before-completion)
- **阶段2**: Web应用功能验证 (ready-to-use-verification)
- **阶段3**: 深度测试 (webapp-testing)

**何时使用**：
- 完成任何 Web 应用开发后
- 用户要求"启动项目"、"部署"之前
- 任何即将呈现工作成果给用户的时刻

### 2. ready-to-use-verification
**开箱即用验证** - 浏览器功能测试

**核心流程**：
1. 启动开发服务器
2. 使用有头浏览器打开（必须能看到浏览器窗口）
3. 验证首页和所有导航链接
4. 测试核心 CRUD 功能
5. 检查浏览器控制台无错误

**铁律**：
```
NO "READY" CLAIMS WITHOUT HEADFUL BROWSER VERIFICATION
```

### 3. verification-before-completion
**完成前验证** - 通用验证原则

**核心原则**：Evidence before claims, always（证据优先于声明）

**验证流程**：
1. IDENTIFY: 什么命令能证明这个声明？
2. RUN: 执行完整的命令
3. READ: 读取完整输出
4. VERIFY: 输出是否确认声明？
5. ONLY THEN: 只有这样才能做出声明

### 4. webapp-testing
**Web 应用测试** - Playwright 自动化测试工具包

**主要功能**：
- Playwright 脚本编写
- 服务器生命周期管理
- DOM 检查和元素发现
- 控制台日志捕获

## 🎯 使用方法

### 完整验证流程（推荐）

```bash
# 在完成开发后，自动触发 self-evolution-system
# 它会依次执行三个阶段的验证
```

### 单独使用

```bash
# 只进行开箱即用验证
skill: ready-to-use-verification

# 只进行代码级验证
skill: verification-before-completion

# 只进行自动化测试
skill: webapp-testing
```

## 🔧 实际案例

### 案例1：订单管理系统

**发现并修复的问题**：
1. `app/customers/page.tsx`: 客户端组件错误使用 async
2. `app/error.tsx`: 重复的 aria-label 属性
3. `app/reports/shipments/users/page.tsx`: aria-level 类型错误
4. `components/ui/Modal.tsx`: 重复属性和混乱的 className
5. `components/ui/Toast.tsx`: 重复的 aria-label
6. `lib/store.ts`: StoreState 未导出、persist 配置错误

**结果**：8个问题被自动发现并修复，用户看到的是完全可用的系统。

## ⚡ 核心原则

1. **证据优先**: 不相信"应该可以"，只相信实际测试结果
2. **用户视角**: 从用户角度测试，不是开发者角度
3. **完整性**: 测试所有声称可用的功能
4. **诚实**: 发现问题如实报告，不隐瞒
5. **迭代**: 发现问题 → 修复 → 重新验证 → 呈现

## 🚫 禁止行为

❌ **不要**:
- 只检查代码就说"ready"
- 只看首页就说"完成"
- 用无头浏览器就说"测试过了"
- 跳过任何验证阶段
- 隐瞒发现的问题

✅ **要**:
- 完整执行三个阶段
- 使用有头浏览器
- 测试所有功能
- 保留截图证据
- 发现问题立即修复

## 💡 时间估算

| 项目规模 | 阶段1 | 阶段2 | 阶段3 | 总计 |
|---------|------|------|------|------|
| 小型 | 2-3分钟 | 5-10分钟 | 5分钟 | 12-18分钟 |
| 中型 | 5分钟 | 15-20分钟 | 10分钟 | 30-35分钟 |
| 大型 | 10分钟 | 30-40分钟 | 20分钟 | 60-70分钟 |

**原则**: 宁可多花 30 分钟验证，不要让用户发现 1 个问题。
