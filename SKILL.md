name: self-evolution-system
description: 基于 autoresearch 理念的文件驱动自主开发系统。AI 通过文件（PRD.md → task.md → progress.md → evaluation.md）沟通协作，每次从 task.md 获取一个任务开发，独立评估器打分，加分确认完成，减分直接重写，迭代直到95分。源自 Andrej Karpathy 的 autoresearch 项目思想。适用于创建新项目或重构现有项目。核心：AI 之间用文件沟通不依赖上下文，换 AI 看着文件也能继续。确保零404、零按钮失效、零占位符。

**何时使用**：
- "帮我开发一个XXX项目"
- "用 self-evolution-system 从零构建项目"
- "重构这个项目"
- "基于 PRD 开发项目"
- 任何需要从需求到完成的项目开发场景

**核心流程**：
阶段1：需求确认 → 生成 PRD.md
阶段2：任务拆解 → PRD.md → task.md
阶段3：自主迭代 → 从 task.md 取任务 → 开发 → 评估打分 → 加分确认/减分重写
阶段4：交付标准 → 达到95分后完成

**关键特性**：
- 文件驱动：AI 之间通过文件沟通，不依赖上下文
- 迭代改进：加分确认完成，减分直接重写
- 独立评估：独立 agent 评估打分，结果写 evaluation.md
- ERROR_LOG：第0轮必读，避免重复错误
- 95分标准：达到95分真实功能后交付

**禁止**：
- 禁止用上下文传递信息（必须用文件）
- 禁止保留减分的代码（直接重写）
- 禁止跳过 ERROR_LOG.md 检查
- 禁止在中间分数询问用户（持续迭代到95分）

# 自我迭代系统 (Self-Evolution System for Development Projects)

## 理念来源

**基于 [autoresearch](https://github.com/karpathy/autoresearch)** (Andrej Karpathy) 的核心思想：

> "Research is now entirely the domain of autonomous swarms of AI agents"

**改造为文件驱动的自主开发系统**：

> "Development is now entirely the domain of autonomous AI agents communicating through files"

### autoresearch vs self-evolution-system

| 维度 | autoresearch | self-evolution-system |
|------|-------------|----------------------|
| 领域 | 机器学习研究 | 通用开发项目 |
| 方法 | AI 修改 train.py → 训练 → 评估 | AI 读取 task.md → 开发 → 评估 |
| 沟通 | 内存变量 | **文件系统** |
| 评估指标 | val_bpb | 0-100分综合评分 |
| 迭代策略 | 保留或丢弃 | 加分确认，减分重写 |
| 停止条件 | val_bpb 最低 | 95分真实功能 |

---

## 核心使命

**AI 通过文件自主协作，从需求到交付，持续迭代直到95分。**

### 关键原则

1. **文件驱动沟通** ⚠️ **最高优先级**
   - AI 之间通过文件沟通，不依赖上下文
   - 换个 AI 看着文件也能继续任务
   - 始终有任务计划（task.md）和开发进度（progress.md）

2. **迭代改进**
   - 加分 → 确认完成，更新 progress.md
   - 减分 → 直接重写，不保留
   - 持续迭代直到95分

3. **独立评估**
   - 独立 agent 评估代码和内容完成度
   - 评估结果写 evaluation.md
   - 包含打分和修改意见

## 文件结构

```
project/
├── PRD.md              # 需求文档（阶段1生成）
├── task.md             # 任务列表（阶段2从PRD拆解）
├── progress.md         # 开发进度（实时更新）
├── evaluation.md       # 评估结果和修改意见（每次评估后更新）
├── ERROR_LOG.md        # 错误历史（第0轮必读）
└── [项目代码]          # 实际开发的项目文件
```

### 文件说明

#### PRD.md（需求文档）
```markdown
# 项目名称 - Product Requirements Document

## 项目概述
- 项目目标
- 目标用户
- 核心功能

## 技术栈
- 前端框架
- 后端技术
- 数据库

## 数据模型
[核心数据结构]

## 页面结构
[所有页面列表]

## 功能列表
[按优先级排序]

## 验收标准
- 功能完整性：60分
- 设计系统：20分
- 代码质量：10分
- 功能验证：10分
- 目标：≥95分
```

```markdown
## 任务拆解原则

## 任务格式

**文件**: [涉及哪些文件]
**验收**: [如何确认完成]

**分数**: [预计贡献分数]

## 任务列表
[按优先级和依赖关系排序]
```

## 开发流程

### 阶段1：需求确认（生成 PRD.md）

**触发**：用户说"帮我开发一个XXX项目"

**步骤**：
1. 询问用户需求（项目类型、核心功能、目标用户）
2. 生成 PRD.md
3. 确认 PRD.md 内容

**输出**：
- PRD.md 文件

---

### 阶段2：任务拆解（PRD.md → task.md）

**触发**：PRD.md 确认后

**步骤**：
1. 读取 PRD.md
2. 拆解成具体任务（task.md）
3. 按优先级和依赖关系排序

**任务拆解原则**：
- 每个任务独立可执行
- 每个任务有明确的验收标准
- 每个任务有预计贡献分数
- 依赖关系清晰

**输出**：
- task.md 文件
- 初始化 progress.md（分数：0/100）

---

### 阶段3：自主迭代开发

**核心循环**：
```
┌─────────────────────────────────────┐
│ 1. 读取 task.md、progress.md        │
│ 2. 获取下一个待办任务               │
│ 3. 开发任务                         │
│ 4. 请求评估（独立agent）            │
│ 5. 读取 evaluation.md               │
│ 6. 如果加分 → 确认完成，更新progress │
│    如果减分 → 直接重写，回到步骤3   │
│ 7. 重复直到95分                     │
└─────────────────────────────────────┘
```

#### 第0轮：ERROR_LOG.md 检查 ⚠️ **最高优先级**

**每次开发新任务前必须执行**：

1. 读取 ERROR_LOG.md
2. 检查是否有类似问题
3. 如果有，先避免再开发
4. 如果发现新问题，记录到 ERROR_LOG.md

**为什么最高优先级**：
- 避免重复错误
- ERROR_LOG.md 是"活文档"，每次都要读

#### 开发单个任务

**步骤**：
1. 读取 task.md，获取下一个待办任务
2. 读取 ERROR_LOG.md，避免重复错误
3. 开发任务（写代码/修改文件）
4. 请求评估（调用独立评估 agent）
5. 读取 evaluation.md，查看分数和意见

**如果加分**：
- ✅ 确认完成
- 更新 task.md（状态：已完成）
- 更新 progress.md（记录完成，更新分数）
- 继续下一个任务

**如果减分**：
- ❌ 直接重写（git reset 或直接删除代码）
- 回到步骤3，重新开发
- 不更新 task.md 和 progress.md
- 再次请求评估

**如果不变**：
- ➡️ 继续下一个任务
- 更新 task.md（状态：已完成）
- 更新 progress.md（记录完成，分数不变）

#### 独立评估（评估 agent）

**评估 agent 职责**：
1. 读取项目文件
2. 检查代码和内容完成度
3. 按照4个维度打分：
   - 功能完整性（60分）
   - 设计系统（20分）
   - 代码质量（10分）
   - 功能验证（10分）
4. 写入 evaluation.md：
   - 当前分数
   - 分数变化
   - 详细评分
   - 修改意见
   - 下一步建议

**评估标准详见**：`references/evaluation-standard.md`

---

### 阶段4：交付

**触发条件**：
- 当前分数 ≥ 95
- 所有 P0 任务完成

**交付检查**：
1. ✅ 零404错误
2. ✅ 零按钮失效
3. ✅ 零占位符
4. ✅ 浏览器实际测试通过
5. ✅ ERROR_LOG.md 中所有错误已解决

**交付输出**：
- 最终 evaluation.md（≥95分）
- 完整的项目代码
- ERROR_LOG.md（所有错误已解决）
- progress.md（完整的开发记录）

### 快速示例

```
用户: "帮我开发一个Vue 3电商项目"

AI阶段1: 生成 PRD.md
  - 询问需求
  - 生成 PRD.md
  - 确认内容

  - 读取 PRD.md
  - 生成 task.md（30个任务）
  - 初始化 progress.md（0/100）

AI阶段3: 自主迭代
  循环开始：
    读取 task.md → 任务#001（初始化项目，+5分）
    读取 ERROR_LOG.md → 无错误
    开发任务#001 → npm create vite@latest
    请求评估 → 评估 agent 打分
    读取 evaluation.md → 5/100，加分 ✅
    更新 progress.md → 5/100，任务#001完成
    更新 task.md → 任务#001状态：已完成

    读取 task.md → 任务#002（添加基础布局，+10分）
    读取 ERROR_LOG.md → 无错误
    开发任务#002 → 创建 Layout 组件
    请求评估 → 评估 agent 打分
    读取 evaluation.md → 12/100，加分 ✅
    更新 progress.md → 12/100，任务#002完成

    ...

    读取 task.md → 任务#015（添加产品列表，+15分）
    读取 ERROR_LOG.md → 发现CSS颜色对比度错误
    避免错误 → 使用正确的颜色配置
    开发任务#015 → 创建 ProductList 组件
    请求评估 → 评估 agent 打分
    读取 evaluation.md → 45/100，减分 ❌
    直接重写 → 删除代码，重新开发
    请求评估 → 评估 agent 打分
    读取 evaluation.md → 62/100，加分 ✅
    更新 progress.md → 62/100，任务#015完成

    ...

    读取 progress.md → 95/100
    检查交付条件 → ✅ 满足
    进入阶段4

AI阶段4: 交付
  - 最终评估确认
  - 交付项目
  - 总结开发记录
```
