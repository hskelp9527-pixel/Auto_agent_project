# 自我迭代系统 (Self-Evolution System for Development Projects)

## 理念来源

**基于 [autoresearch](https://github.com/karpathy/autoresearch)** (Andrej Karpathy) 的核心思想：

> "Research is now entirely the domain of autonomous swarms of AI agents"

**改造为通用开发项目 QA 领域**：

> "Quality Assurance is now entirely the domain of autonomous AI agents before user sees it"

**最佳应用场景**：
- ✅ Web 应用（最佳适配）
- ✅ 移动应用
- ✅ 桌面应用
- ✅ 后端服务
- ✅ CLI 工具
- ✅ 任意开发类项目

## 核心使命

**在用户看到项目之前，发现问题并修复。**

不要让用户成为你的测试员。在说"完成了"、"ready"、"可以启动了"之前，必须完成自我迭代验证流程。

## 何时触发

**必须使用此系统的情况**：
- 完成任何开发项目后（Web/移动/桌面/后端等）
- 修复重要 bug 并准备提交后
- 添加新功能并准备展示后
- 用户要求"启动项目"、"部署"、"可以用了"、"测试一下"、"评估质量"之前
- 任何即将呈现工作成果给用户的时刻

**触发关键词**：
- "完成了"
- "ready"
- "可以启动了"
- "给用户看"
- "准备部署"
- "项目做完了"
- "验收"
- "测试一下"
- "评估质量"

## 验证流程概述

本系统采用至少5轮迭代验证：

1. **第0轮：错误历史检查** ⚠️ **最高优先级** → 详见 `references/round-0-error-history.md`
2. **第1轮：内容完整性检查** ⭐ **60%** → 详见 `references/round-1-content-completeness.md`
3. **第2轮：设计系统质量检查** ⭐ **20%** → 详见 `references/round-2-design-system.md`
4. **第3轮：代码质量检查** ⭐ **10%** → 详见 `references/round-3-code-quality.md`
5. **第4轮：浏览器功能验证** ⭐ **10%** → 详见 `references/round-4-browser-verification.md`

根据项目实际情况可增加第5+轮专项验证（如性能优化、安全检查、兼容性等）。

## 快速开始

### 验证流程

```bash
# 第1步：读取当前SKILL.md
# 第2步：依次读取并执行各轮验证
# 第3步：计算总分并判断是否通过
```

**执行顺序**：
1. **必须**从第0轮开始（ERROR_LOG.md检查）
2. 依次执行第1-4轮
3. 每轮不通过必须重新该轮
4. 所有轮次通过后计算总分
5. 达到95分真实功能后才能交付

### 评分标准

```
总分 = 第0轮(关键项) + 第1轮(60分) + 第2轮(20分) + 第3轮(10分) + 第4轮(10分) = 100+分
```

**通过标准**：
- **优秀**：≥ 95分 - 可以自信地交付给用户
- **良好**：≥ 80分 - 基本可用，有小瑕疵
- **不通过**：< 80分 - 存在重大问题，必须修复

## 决策树

```
开始
  ↓
第0轮：ERROR_LOG检查（最高优先级）
  ↓ 通过？→ 否 → 修复问题 → 重新第0轮
  ↓ 是
第1轮：内容完整性（60分）
  ↓ 通过？→ 否 → 修复问题 → 重新第1轮
  ↓ 是
第2轮：设计系统（20分）
  ↓ 通过？→ 否 → 修复问题 → 重新第2轮
  ↓ 是
第3轮：代码质量（10分）
  ↓ 通过？→ 否 → 修复问题 → 重新第3轮
  ↓ 是
第4轮：浏览器验证（10分）
  ↓ 通过？→ 否 → 修复问题 → 重新第4轮
  ↓ 是
计算总分 ≥ 95？
  ↓ 否 → 修复剩余问题 → 交付
  ↓ 是
是否还有未解决问题？
  ↓ 是 → 增加专项验证轮（第5+轮）→ 重新验证
  ↓ 否
直接交付 🎉
```

**重要说明**：
- ✅ 前4轮是基础验证，适用于所有项目
- ✅ 第5+轮根据项目实际情况灵活添加
- ✅ 不是固定4轮，而是**至少4轮**
- ✅ 每轮不通过必须重新该轮（零容忍）

## 关键原则与常见陷阱

完整的关键原则和常见陷阱请参考：`references/key-principles.md`

**核心原则摘要**：
1. 零容忍原则（404错误、按钮失效、占位符）
2. 用户体验优先（功能可用、页面可访问、交互有反馈）
3. 迭代改进（不通过必须重新、不能跳过、彻底修复）
4. 真实验证（浏览器测试、不能只看代码）
5. **ERROR_LOG.md强制使用**（最高优先级）
6. **持续验证至95分ACTUAL功能**（静态≠真实）
7. **非必要不询问**（禁止中间分数询问）
8. **CSS颜色对比度强制检查**
9. **UI可读性全面验证**

## 最佳实践

✅ **每轮迭代都记录问题** - 便于追踪改进
✅ **优先修复核心问题** - 从第0轮开始
✅ **使用有头浏览器验证** - 确保真实可用
✅ **用户测试前自测** - 不要让用户当测试员
✅ **达到95分再交付** - 确保高质量

## 📖 理念来源

本系统基于 [autoresearch](https://github.com/karpathy/autoresearch) (Andrej Karpathy) 的核心思想设计。

**autoresearch 原版**：
- 领域：机器学习研究
- 方法：AI 修改 `train.py` → 训练 5 分钟 → 评估 → 保留或丢弃
- 目标：优化模型性能（降低 val_bpb）

**本系统改造**：
- 领域：任意开发类项目（Web/移动/桌面/后端等）
- 方法：多轮迭代验证（至少4轮，根据需要增加第5+轮）
- 目标：确保上线前质量（零 bug、零 404、功能完整）
- 最佳适配：Web 应用等 UI 可见项目

**核心共同点**：
- ✅ AI 自主迭代
- ✅ 基于证据的决策
- ✅ 持续改进直到达到标准
- ✅ 完整的实验历史和追踪

**关键差异**：
- autoresearch：自动训练，评估指标是 val_bpb
- 本系统：手动验证，评估标准是多维度评分（100分制）
- autoresearch：无人值守持续优化
- 本系统：每轮不通过必须重新该轮（零容忍）
