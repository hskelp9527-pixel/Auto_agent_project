# 自我迭代系统 (Self-Evolution System)

## 核心使命

**在用户看到项目之前，发现问题并修复。**

不要让用户成为你的测试员。在说"完成了"、"ready"、"可以启动了"之前，必须完成自我迭代验证流程。

## 何时触发

**必须使用此系统的情况**：
- 完成任何 Web 应用开发后
- 修复重要 bug 并准备提交后
- 添加新功能并准备展示后
- 用户要求"启动项目"、"部署"、"可以用了"之前
- 任何即将呈现工作成果给用户的时刻

**触发关键词**：
- "完成了"
- "ready"
- "可以启动了"
- "给用户看"
- "准备部署"
- "项目做完了"

## 自我迭代验证流程

### 第1轮迭代：内容完整性检查 (content-completeness-check) ⭐ **核心权重60%**

**目标**: 确保所有功能都有实际内容，不是空壳

**为什么这是第1轮？**
- **功能完整性是核心**，权重占60%
- 内容缺失会让用户立即判断项目"未完成"
- 设计再好，没有内容也是零分
- 这是用户最先看到的东西

**执行步骤**：

1. **页面完整性检查**
   ```bash
   # 检查每个页面是否有实际内容
   # 方法1：检查所有页面文件
   find app -name "page.tsx" -exec sh -c 'echo "=== $1 ===" && grep -E "占位|TODO|暂无|empty|placeholder" "$1"' _ {} \;

   # 方法2：检查是否有占位符文本
   grep -r "订单表单\|库存列表\|数据加载中\|待开发" app/ --include="*.tsx"
   ```

   **必须避免的占位符**：
   - ❌ 4字占位符（如"订单表单"、"库存列表"）
   - ❌ "暂无数据"、"数据加载中"
   - ❌ "TODO"、"待开发"、"coming soon"
   - ❌ 空白组件（`<p>内容</p>`、`<div></div>`）
   - ❌ 只有标题没有内容

2. **表单完整性检查**
   ```bash
   # 检查所有创建/编辑表单页面
   find app -path "*/create/page.tsx" -o -path "*/edit/page.tsx"

   # 检查表单是否包含必要元素
   grep -r "input\|select\|textarea\|form" app/*/create/page.tsx
   ```

   **表单最低要求**：
   - ✅ 至少有3个以上输入字段
   - ✅ 有提交/保存按钮
   - ✅ 有必填字段标识
   - ✅ 有表单验证
   - ✅ 有取消/返回按钮

3. **示例数据检查**
   ```bash
   # 检查是否有足够的示例数据
   grep -r "mockData\|EXAMPLE\|const.*=.*\[" lib/ data/

   # 统计数据条数
   grep -r "export const.*:" lib/ | grep -o "\[.*\]" | head -5
   ```

   **最少数据要求**：
   - ✅ 每个主要实体至少5-10条数据
   - ✅ 数据字段完整（不是空对象）
   - ✅ 数据有真实感（不是"测试1"、"测试2"）
   - ✅ 关联数据之间有逻辑关系

4. **图表完整性检查**（如适用）
   ```bash
   # 检查是否安装并使用了图表库
   cat package.json | grep -E "recharts|chartjs|vis|echarts"
   grep -r "from.*chart" app/ --include="*.tsx"
   ```

   **图表最低要求**（如果有数据展示页面）：
   - ✅ 至少有2个不同类型的图表
   - ✅ 图表有真实数据源
   - ✅ 图表有标题和图例
   - ✅ 图表能正常渲染

5. **CRUD功能检查**
   ```bash
   # 检查是否有增删改查功能
   grep -r "create\|add\|edit\|update\|delete\|remove" app/ --include="*.tsx" | grep -i "button\|link"

   # 检查store是否有对应方法
   grep -r "add\|update\|delete" lib/store.ts
   ```

   **必须具备**：
   - ✅ 新建/添加按钮和页面
   - ✅ 编辑功能（每个主要实体）
   - ✅ 删除功能（每个主要实体）
   - ✅ 查看详情功能
   - ✅ 功能连接到数据store

6. **操作按钮检查**
   ```bash
   # 检查表格列表页是否有操作列
   grep -r "操作\|Actions\|action" app/ --include="*.tsx" -A 2
   ```

   **必须包含**：
   - ✅ 编辑按钮
   - ✅ 删除按钮
   - ✅ 查看详情按钮
   - ✅ 按钮有明确的图标或文字

7. **功能可用性检查** ⚠️ **关键项**
   ```bash
   # 检查按钮是否有实际功能处理
   # 1. 检查是否有onClick处理函数
   grep -r "onClick.*handle\|onClick.*\(" app/ --include="*.tsx" | grep -E "delete|remove|edit"

   # 2. 检查删除功能是否连接到store
   grep -r "delete\|remove" app/ --include="*.tsx" | grep -v "className\|aria-label"

   # 3. 检查详情页路由是否存在
   find app -path "*/[id]/page.tsx"
   ```

   **必须具备**：
   - ✅ 删除按钮有onClick处理函数
   - ✅ 删除功能调用store的delete方法
   - ✅ 删除前有确认对话框（confirm）
   - ✅ 查看按钮链接到实际存在的详情页路由
   - ✅ 编辑按钮链接到编辑页或触发编辑功能
   - ✅ 所有按钮都可以实际点击并执行操作

   **常见错误**：
   - ❌ 按钮存在但无onClick处理（点击无反应）
   - ❌ 按钮有onClick但未连接到store（点击无效）
   - ❌ 链接指向不存在的路由（404错误）
   - ❌ 删除操作无确认对话框（误删风险）

8. **路由存在性检查** ⚠️ **零容忍项**
   ```bash
   # 检查所有按钮链接的页面是否都存在
   # 1. 提取所有Link和按钮的href/target
   grep -r "href=\|to=" app/ --include="*.tsx" | grep -o "/[^\"]*" | sort -u

   # 2. 检查编辑页是否存在
   find app -path "*/[id]/edit/page.tsx"

   # 3. 检查详情页是否存在
   find app -path "*/[id]/page.tsx"

   # 4. 检查创建页是否存在
   find app -path "*/create/page.tsx"
   ```

   **零容忍规则**：
   - ✅ **所有按钮链接的页面都必须存在**
   - ✅ **绝不出现404错误**
   - ✅ **哪怕功能不齐全，页面也必须能打开**
   - ✅ **每个主要实体至少有：列表页、详情页、编辑页**

   **最低页面要求**（每个实体）：
   - ✅ 列表页：显示数据列表
   - ✅ 详情页：显示单条数据详情
   - ✅ 编辑页：编辑数据
   - ✅ 创建页：新建数据（核心实体）

   **常见404错误模式**：
   - ❌ `/[entity]/[id]/edit` - 编辑页不存在
   - ❌ `/[entity]/[id]` - 详情页不存在
   - ❌ `/[entity]/create` - 创建页不存在
   - ❌ 链接href拼写错误

   **评分标准**：
   - **发现一个404，功能可用性5分全部扣除**
   - **即使总分60分，出现404也视为不通过**

**内容完整性评分**（总分60分）：
- 页面完整性：18分
- 表单完整性：13分
- 示例数据：10分
- 图表完整性：10分
- CRUD功能：4分
- **功能可用性：5分** ⭐ 包含路由存在性

**通过标准**：≥ 48分（80%）

**不通过**：必须补充内容或修复功能，重新进入第1轮迭代

**特别注意**：
- 功能可用性是关键项（5分），如果按钮不可用或存在404，直接扣除全部5分
- 即使有60分，如果功能不可用或存在404，也视为不通过
- 零容忍：绝不出现404错误

### 第2轮迭代：设计系统质量检查 (design-system-check) ⭐ **权重20%**

**目标**: 确保项目具备企业级设计系统标准

**为什么要检查设计系统？**
- 功能完整 ≠ 设计完整
- 设计系统质量直接影响用户体验
- 企业级应用必须具备专业设计系统
- 避免"功能强大但视觉效果差"的问题

**执行步骤**：

1. **核心依赖检查**
   ```bash
   # 检查是否具备现代化UI组件库生态
   cat package.json | grep -E "class-variance-authority|tailwind-merge|@radix-ui|@headlessui"
   ```

   **必需依赖**：
   - ✅ `class-variance-authority` (CVA)
   - ✅ `tailwind-merge`
   - ✅ 至少一个无障碍组件库（Radix UI / Headless UI）

2. **组件变体系统检查**
   ```bash
   # 检查是否使用CVA创建组件变体
   grep -r "cva\|variants" components/ --include="*.tsx" -A 3
   ```

   **必须具备**：
   - ✅ Button组件至少有3个变体（default/outline/ghost）
   - ✅ 至少有3个基础组件使用CVA
   - ✅ 变体包含size属性（sm/md/lg）
   - ✅ 变体包含视觉属性（color/variant）

3. **设计令牌检查**
   ```bash
   # 检查是否有设计令牌系统
   cat tailwind.config.ts | grep -A 20 "colors:\|theme"

   # 检查是否有CSS变量定义
   grep -r "--.*-" app/globals.css -A 2
   ```

   **必须具备**：
   - ✅ 至少定义了5个基础颜色（primary/secondary/success/warning/danger）
   - ✅ 颜色有不同深浅（50-950）
   - ✅ 有CSS变量系统（light/dark模式）
   - ✅ 至少20个CSS变量

4. **无障碍支持检查**
   ```bash
   # 检查组件是否有无障碍属性
   grep -r "aria-\|role=" components/ app/ --include="*.tsx"
   ```

   **必须具备**：
   - ✅ 按钮有aria-label
   - ✅ 表单输入有关联的label
   - ✅ 图标有aria-hidden="true"
   - ✅ 交互元素有role属性

5. **响应式设计检查**
   ```bash
   # 检查是否有响应式断点
   grep -r "md:\|lg:\|xl:" components/ app/ --include="*.tsx"
   ```

   **必须具备**：
   - ✅ 主要布局有移动端适配
   - ✅ 表格在小屏幕可横向滚动
   - ✅ 导航在移动端可用
   - ✅ 至少使用3个断点（md/lg/xl）

**设计系统评分**（总分20分）：
- 核心依赖：5分
- 组件变体系统：5分
- 设计令牌：5分
- 无障碍支持：3分
- 响应式设计：2分

**通过标准**：≥ 16分（80%）

**不通过**：必须补充设计系统，重新进入第2轮迭代

### 第3轮迭代：代码质量检查 (code-quality-check) ⭐ **权重10%**

**目标**: 确保代码规范、健壮、可维护

**为什么要检查代码质量？**
- 好的代码质量决定项目可维护性
- 避免技术债务积累
- 确保长期可扩展性

**执行步骤**：

1. **TypeScript使用检查**
   ```bash
   # 检查是否使用TypeScript
   find . -name "*.ts" -o -name "*.tsx" | wc -l

   # 检查是否有any类型
   grep -r ": any" app/ components/ --include="*.tsx" --include="*.ts"
   ```

   **必须具备**：
   - ✅ 至少90%代码使用TypeScript
   - ✅ `any`类型使用少于5处
   - ✅ 组件有Props类型定义

2. **组件化检查**
   ```bash
   # 检查是否有可复用组件
   ls components/

   # 检查组件复用情况
   grep -r "from.*components" app/ --include="*.tsx" | wc -l
   ```

   **必须具备**：
   - ✅ 至少有5个可复用组件
   - ✅ 组件在多处被引用
   - ✅ 组件职责单一

3. **错误处理检查**
   ```bash
   # 检查是否有错误处理
   grep -r "try\|catch\|error" app/ --include="*.tsx"
   ```

   **必须具备**：
   - ✅ 表单提交有错误处理
   - ✅ API调用有错误处理
   - ✅ 用户友好的错误提示

4. **代码规范检查**
   ```bash
   # 检查是否有代码格式化配置
   ls .prettierrc .eslintrc*

   # 检查代码一致性
   grep -r "import.*from" app/ --include="*.tsx" | head -10
   ```

   **必须具备**：
   - ✅ 有代码格式化配置（Prettier/ESLint）
   - ✅ import语句规范有序
   - ✅ 命名规范一致

**代码质量评分**（总分10分）：
- TypeScript使用：3分
- 组件化：3分
- 错误处理：2分
- 代码规范：2分

**通过标准**：≥ 8分（80%）

**不通过**：必须优化代码质量，重新进入第3轮迭代

### 第4轮迭代：浏览器功能验证 (browser-verification) ⭐ **权重10%**

**目标**: 确保所有功能在浏览器中实际可用

**为什么要进行浏览器验证？**
- 代码不等于功能
- 只有实际运行才能发现隐藏问题
- 用户最终使用的是浏览器中的应用

**执行步骤**：

1. **启动服务器检查**
   ```bash
   # 启动开发服务器
   npm run dev

   # 检查是否成功启动
   # 应该看到：Ready in Xs, Local: http://localhost:XXXX
   ```

   **必须确保**：
   - ✅ 服务器成功启动无报错
   - ✅ 无TypeScript编译错误
   - ✅ 无依赖缺失错误

2. **页面导航检查**
   ```bash
   # 使用agent-browser或手动检查所有主要页面
   # 1. 首页
   # 2. 列表页（每个实体）
   # 3. 详情页（每个实体）
   # 4. 创建/编辑页
   ```

   **必须确保**：
   - ✅ 所有链接可点击
   - ✅ 所有页面可访问（无404）
   - ✅ 页面加载正常
   - ✅ 无控制台错误

3. **功能交互检查**
   ```bash
   # 测试主要功能
   # 1. 创建新数据
   # 2. 编辑数据
   # 3. 删除数据
   # 4. 搜索/筛选
   ```

   **必须确保**：
   - ✅ 表单可提交
   - ✅ 按钮点击有响应
   - ✅ 数据可保存
   - ✅ 状态正确更新

4. **响应式测试**
   ```bash
   # 测试不同屏幕尺寸
   # 1. 桌面（1920x1080）
   # 2. 平板（768x1024）
   # 3. 移动（375x667）
   ```

   **必须确保**：
   - ✅ 所有尺寸可用
   - ✅ 布局正确适配
   - ✅ 触摸操作可用

**浏览器功能评分**（总分10分）：
- 服务器启动：2分
- 页面导航：3分
- 功能交互：3分
- 响应式：2分

**通过标准**：≥ 8分（80%）

**不通过**：必须修复问题，重新进入第4轮迭代

## 总体评分与决策

### 综合评分计算

```
总分 = 第1轮(60分) + 第2轮(20分) + 第3轮(10分) + 第4轮(10分) = 100分
```

### 通过标准

**优秀**：≥ 95分
- 可以自信地交付给用户
- 已达到企业级应用标准

**良好**：≥ 80分
- 基本可用，有小瑕疵
- 建议修复后再交付

**不通过**：< 80分
- 存在重大问题
- 必须修复后才能交付

### 决策树

```
开始
  ↓
第1轮迭代（内容完整性 60分）
  ↓ 通过？→ 否 → 修复问题 → 重新第1轮
  ↓ 是
第2轮迭代（设计系统 20分）
  ↓ 通过？→ 否 → 修复问题 → 重新第2轮
  ↓ 是
第3轮迭代（代码质量 10分）
  ↓ 通过？→ 否 → 修复问题 → 重新第3轮
  ↓ 是
第4轮迭代（浏览器功能 10分）
  ↓ 通过？→ 否 → 修复问题 → 重新第4轮
  ↓ 是
计算总分 ≥ 95？
  ↓ 否
修复剩余问题 → 交付
  ↓ 是
直接交付 🎉
```

## 关键原则

1. **零容忍原则**
   - 404错误零容忍
   - 按钮无反应零容忍
   - 占位符内容零容忍

2. **用户体验优先**
   - 功能必须可用，不能只是摆设
   - 所有页面必须能访问
   - 所有交互必须有反馈

3. **迭代改进**
   - 每轮不通过必须重新该轮
   - 不允许跳过任何检查
   - 问题必须彻底修复

4. **真实验证**
   - 必须在浏览器中实际测试
   - 不能只看代码
   - 使用有头浏览器验证关键功能

## 常见陷阱

❌ **只看代码不运行** - 必须在浏览器中验证
❌ **只检查功能不检查设计** - 设计系统同样重要
❌ **只检查新增不检查整体** - 要确保整体质量
❌ **发现小问题就跳过** - 所有问题必须修复
❌ **相信"暂不实现"** - 临时方案往往是永久方案

## 最佳实践

✅ **每轮迭代都记录问题** - 便于追踪改进
✅ **优先修复核心问题** - 从第1轮开始
✅ **使用有头浏览器验证** - 确保真实可用
✅ **用户测试前自测** - 不要让用户当测试员
✅ **达到95分再交付** - 确保高质量

---

**记住**：这个系统的目标是在用户看到项目之前，发现并修复所有问题。不要让用户成为你的测试员！
