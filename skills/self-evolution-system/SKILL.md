# 自我迭代系统 (Self-Evolution System)

## 核心使命

**在用户看到项目之前，发现问题并修复。**

不要让用户成为你的测试员。在说"完成了"、"ready"、"可以启动了"之前，必须完成自我迭代验证流程。

## 何时触发

**必须使用此系统的情况**：
- 完成任何 Web 应用开发后
- 修复重要 bug 并准备提交后
- 添加新功能并准备展示后
- 用户要求"启动项目"、"部署"、"可以用了"之前
- 任何即将呈现工作成果给用户的时刻

**触发关键词**：
- "完成了"
- "ready"
- "可以启动了"
- "给用户看"
- "准备部署"
- "项目做完了"

## 自我迭代验证流程

### 第1轮迭代：内容完整性检查 (content-completeness-check) ⭐ **核心权重60%**

**目标**: 确保所有功能都有实际内容，不是空壳

**为什么这是第1轮？**
- **功能完整性是核心**，权重占60%
- 内容缺失会让用户立即判断项目"未完成"
- 设计再好，没有内容也是零分
- 这是用户最先看到的东西

**执行步骤**：

1. **页面完整性检查**
   ```bash
   # 检查每个页面是否有实际内容
   for page in orders products inventory finance customers reports; do
     echo "=== 检查 $page 页面 ==="
     cat app/$page/page.tsx | grep -E "占位|TODO|暂无|empty" && echo "⚠️ 发现占位符"
   done
   ```

   **必须避免的占位符**：
   - ❌ "订单表单"（四个字就没了）
   - ❌ "库存列表"（四个字就没了）
   - ❌ "暂无数据"
   - ❌ "TODO"
   - ❌ `<p>内容</p>`

2. **表单完整性检查**
   ```bash
   # 检查关键表单页面
   cat app/orders/create/page.tsx
   cat app/products/create/page.tsx
   ```

   **必需的表单字段**：

   **新建订单表单**：
   - ✅ 客户选择器（下拉/搜索）
   - ✅ 产品列表（多选，带价格）
   - ✅ 数量输入
   - ✅ 金额计算和显示
   - ✅ 配送地址表单
   - ✅ 订单日期选择
   - ✅ 备注输入
   - ✅ 提交按钮

   **新建产品表单**：
   - ✅ 产品名称
   - ✅ 产品编码
   - ✅ 分类选择
   - ✅ 规格说明
   - ✅ 单位
   - ✅ 价格
   - ✅ 库存数量
   - ✅ 最小库存

3. **示例数据检查**
   ```bash
   # 检查是否有足够的示例数据
   grep -r "mockData\|EXAMPLE" lib/ app/
   ```

   **最少数据要求**：
   - ✅ 订单列表：至少5条
   - ✅ 产品列表：至少10条
   - ✅ 客户列表：至少5条
   - ✅ 库存列表：至少10条
   - ✅ 财务数据：至少3个月

4. **图表完整性检查**
   ```bash
   # 检查是否安装并使用了图表库
   cat package.json | grep "recharts"
   grep -r "from 'recharts'" app/
   ```

   **必需的图表**：
   - ✅ 财务管理：收入趋势图（折线图）
   - ✅ 财务管理：支出分类饼图
   - ✅ 数据报表：销售统计柱状图
   - ✅ 数据报表：产品排行图

5. **CRUD功能检查**
   ```bash
   # 检查是否有增删改查功能
   grep -r "新建\|创建\|编辑\|删除" app/*/page.tsx
   ```

   **必须具备**：
   - ✅ 新建按钮和页面
   - ✅ 编辑功能（每个实体）
   - ✅ 删除功能（每个实体）
   - ✅ 查看详情功能

6. **操作按钮检查**
   ```bash
   # 检查表格是否有操作列
   grep -A5 "操作列\|Actions" app/*/page.tsx
   ```

   **必须包含**：
   - ✅ 编辑按钮
   - ✅ 删除按钮
   - ✅ 查看详情按钮

7. **功能可用性检查** ⚠️ **关键项**
   ```bash
   # 检查按钮是否有实际功能处理
   # 1. 检查是否有onClick处理函数
   grep -r "onClick.*handleDelete\|onClick.*delete" app/*/page.tsx

   # 2. 检查删除功能是否连接到store
   grep -r "deleteOrder\|deleteProduct\|deleteCustomer" app/*/page.tsx lib/store.ts

   # 3. 检查详情页路由是否存在
   ls app/orders/[id]/page.tsx
   ls app/products/[id]/page.tsx
   ls app/customers/[id]/page.tsx
   ```

   **必须具备**：
   - ✅ 删除按钮有onClick处理函数
   - ✅ 删除功能调用store的delete方法
   - ✅ 删除前有确认对话框（confirm）
   - ✅ 查看按钮链接到实际存在的详情页路由
   - ✅ 编辑按钮链接到编辑页或触发编辑功能
   - ✅ 所有按钮都可以实际点击并执行操作

   **常见错误**：
   - ❌ 按钮存在但无onClick处理（点击无反应）
   - ❌ 按钮有onClick但未连接到store（点击无效）
   - ❌ 链接指向不存在的路由（404错误）
   - ❌ 删除操作无确认对话框（误删风险）

8. **路由存在性检查** ⚠️ **零容忍项**
   ```bash
   # 检查所有按钮链接的页面是否都存在
   # 1. 提取所有Link和按钮的href/target
   grep -r "href=.*edit\|to=.*edit" app/ | grep -o "href=\"[^\"]*\"" | sort -u

   # 2. 检查编辑页是否存在
   find app -name "edit" -type d
   find app -path "*/[id]/edit/page.tsx"

   # 3. 检查详情页是否存在
   find app -path "*/[id]/page.tsx"

   # 4. 验证所有必需的路由
   ls app/orders/[id]/page.tsx          # 订单详情
   ls app/orders/[id]/edit/page.tsx     # 订单编辑
   ls app/products/[id]/page.tsx        # 产品详情
   ls app/products/[id]/edit/page.tsx   # 产品编辑
   ls app/customers/[id]/page.tsx       # 客户详情
   ls app/customers/[id]/edit/page.tsx  # 客户编辑
   ```

   **零容忍规则**：
   - ✅ **所有按钮链接的页面都必须存在**
   - ✅ **绝不出现404错误**
   - ✅ **哪怕功能不齐全，页面也必须能打开**
   - ✅ **每个实体至少有：列表页、详情页、编辑页**

   **最低页面要求**：
   - ✅ 订单：列表、详情、编辑、创建
   - ✅ 产品：列表、详情、编辑、创建
   - ✅ 客户：列表、详情、编辑、创建
   - ✅ 库存：列表、详情、编辑
   - ✅ 发货：列表、详情、编辑

   **常见404错误**：
   - ❌ `/orders/[id]/edit` - 订单编辑页不存在
   - ❌ `/products/[id]/edit` - 产品编辑页不存在
   - ❌ `/customers/[id]/edit` - 客户编辑页不存在
   - ❌ `/customers/[id]` - 客户详情页不存在
   - ❌ `/inventory/[id]` - 库存详情页不存在

   **评分标准**：
   - **发现一个404，功能可用性5分全部扣除**
   - **即使总分60分，出现404也视为不通过**

**内容完整性评分**（总分60分）：
- 页面完整性：18分
- 表单完整性：13分
- 示例数据：10分
- 图表完整性：10分
- CRUD功能：4分
- **功能可用性：5分** ⭐ 新增

**通过标准**：≥ 48分（80%）

**不通过**：必须补充内容或修复功能，重新进入第1轮迭代

**特别注意**：
- 功能可用性是关键项（5分），如果按钮不可用，直接扣除全部5分
- 即使有60分，如果功能不可用，也视为不通过

### 第2轮迭代：设计系统质量检查 (design-system-check) ⭐ **权重20%**

**目标**: 确保项目具备电商级设计系统标准

**为什么要检查设计系统？**
- 功能完整 ≠ 设计完整
- 设计系统质量直接影响用户体验
- 电商级应用必须具备企业级设计系统
- 避免"功能强大但视觉效果差"的问题

**执行步骤**：

1. **核心依赖检查**
   ```bash
   # 检查是否具备shadcn/ui生态
   cat package.json | grep -E "class-variance-authority|tailwind-merge|@radix-ui"
   ```

   必备依赖：
   - ✅ class-variance-authority (CVA)
   - ✅ tailwind-merge (className优化)
   - ✅ tailwindcss-animate (动画)
   - ✅ @radix-ui/* (无障碍组件)
   - ✅ next-themes (暗色模式)

2. **Tailwind配置检查**
   ```bash
   # 检查是否有完整的设计令牌
   cat tailwind.config.ts | grep -E "darkMode|hsl\(var\(--"
   ```

   必备配置：
   - ✅ darkMode: ["class"]
   - ✅ HSL颜色变量 (hsl(var(--...)))
   - ✅ 动画插件 (tailwindcss-animate)

3. **CSS变量系统检查**
   ```bash
   # 检查globals.css中的设计令牌
   cat app/globals.css | grep -E "var\(--|:root|\.dark"
   ```

   最少要求：
   - ✅ 亮色主题: 20+个变量
   - ✅ 暗色主题: 20+个变量
   - ✅ @layer base配置

4. **工具函数检查**
   ```bash
   # 检查cn()函数
   cat lib/utils.ts | grep -E "twMerge|cn"
   ```

5. **组件质量检查**
   ```bash
   # 检查Button组件是否使用CVA
   cat components/ui/Button.tsx | grep -E "cva|VariantProps"
   ```

   标准要求：
   - ✅ 使用class-variance-authority
   - ✅ 使用VariantProps类型
   - ✅ 无硬编码颜色
   - ✅ 无className重复

6. **无障碍性检查**
   ```bash
   # 查找常见问题
   grep -r 'aria-label.*aria-label' app/ components/
   ```

   应该为0个结果（无重复ARIA属性）

7. **评分和报告**
   - 优秀 (90-100分): 继续下一阶段
   - 良好 (70-89分): 建议改进后继续
   - 需要改进 (50-69分): 必须修复关键问题
   - 不合格 (<50分): 停止，必须重构设计系统

**退出条件**：
- ✅ 核心依赖齐全
- ✅ CSS变量系统完整
- ✅ 组件使用CVA
- ✅ 评分 ≥ 70分

**时间估算**: 5-10分钟

**目标**: 确保项目具备电商级设计系统标准

**为什么要第2轮检查设计系统？**
- 内容完整后，设计质量决定用户体验
- 权重占20%（比内容低，但仍然重要）
- 设计系统好但内容空壳 = 0分
- 设计系统好 + 内容完整 = 高分

**执行步骤**：（保持原有的设计系统检查步骤）

**设计系统评分**（总分20分）：
- 核心依赖：5分
- Tailwind配置：3分
- CSS变量系统：5分
- 工具函数：2分
- 组件质量：3分
- 无障碍性：5分
- 暗色模式：5分
- **总分31分，按比例折算为20分**

**通过标准**：≥ 14分（70%）

## 迭代决策树

```
开发完成
    ↓
[第1轮迭代] 内容完整性检查 (权重60%) ⭐ 核心
    ├─ ≥ 48分 → 进入第2轮
    ├─ 30-47分 → 补充内容 → 重新第1轮
    └─ < 30分 → 重大内容缺失 → 重新第1轮
         ↓
[第2轮迭代] 设计系统质量检查 (权重20%)
    ├─ ≥ 14分 → 进入第3轮
    ├─ 10-13分 → 修复设计问题 → 重新第2轮
    └─ < 10分 → 重构设计系统 → 重新第2轮
         ↓
[第3轮迭代] 代码级验证 (权重10%)
    ├─ ≥ 7分 → 进入第4轮
    └─ < 7分 → 修复代码问题 → 重新第3轮
         ↓
[第4轮迭代] 浏览器功能验证 (权重10%)
    ├─ ≥ 7分 → ✅ 可以呈现给用户
    └─ < 7分 → 修复功能问题 → 重新第4轮
         ↓
    （可选）[第5轮] 深度测试
```

**关键原则**：
- **第1轮最重要**：内容不完整，直接重新开始
- **每轮必须通过**：不通过就重新该轮
- **明确显示"第X轮"**：让用户看到迭代次数
- **记录迭代轮次**：在第X轮发现问题，第X轮修复

## 常见失败模式和修复

### 第1轮常见问题（内容完整性）
- **页面空白/占位符**: 用户看到"暂无数据"
  - 修复: 补充示例数据（每页至少5-10条）
- **表单缺失**: 新建订单页面只有"订单表单"四个字
  - 修复: 添加完整表单字段
- **图表缺失**: 安装recharts但未使用
  - 修复: 实现图表组件
- **CRUD功能缺失**: 没有编辑/删除功能
  - 修复: 添加操作列和按钮

### 第2轮常见问题（设计系统）
- **缺少shadcn/ui**: 无设计系统标准
  - 修复: `npx shadcn-ui@latest init`
- **无CVA**: variants管理混乱
  - 修复: `npm install class-variance-authority`
- **硬编码颜色**: 无法统一主题
  - 修复: 使用CSS变量 `hsl(var(--primary))`
- **className重复**: 样式冲突
  - 修复: 使用`tailwind-merge`的`cn()`函数
- **无暗色模式**: 用户体验受限
  - 修复: 安装`next-themes`并配置
- **无障碍性差**: ARIA属性错误
  - 修复: 使用Radix UI组件

### 第3轮常见问题（代码级验证）
- **构建失败**: 语法错误、类型错误、依赖问题
- **Linter 错误**: 代码规范问题、未使用变量
- **Import 错误**: 路径错误、模块未导出

### 第4轮常见问题（浏览器功能验证）
- **服务器启动失败**: 端口占用、配置错误
- **页面空白**: 组件错误、数据未加载
- **导航不工作**: 路由配置错误、Link 组件问题
- **控制台错误**: JavaScript 运行时错误、API 调用失败
- **功能缺失**: TODO 未实现、空函数

### 第5轮常见问题（深度测试-可选）
- **用户体验差**: 加载慢、交互卡顿
- **边界情况**: 空数据、错误数据未处理
- **表单问题**: 验证不足、提交失败

| 中型 | 10-15分钟 | 5-8分钟 | 5分钟 | 15-20分钟 | 35-48分钟 |
| 大型 | 15-20分钟 | 8-10分钟 | 10分钟 | 30-40分钟 | 63-80分钟 |

**原则**: 宁可多花时间验证内容完整性（第1轮最重要），不要让用户看到空壳项目。

**说明**：
- 第1轮（内容完整性）最耗时，但最重要
- 第2-4轮相对快速，因为是检查性质
- 如果需要多轮迭代，时间会相应增加

## 核心原则

1. **证据优先**: 不相信"应该可以"，只相信实际测试结果
2. **用户视角**: 从用户角度测试，不是开发者角度
3. **完整性**: 测试所有声称可用的功能
4. **诚实**: 发现问题如实报告，不隐瞒
5. **迭代**: 发现问题 → 修复 → 重新验证 → 呈现

## 禁止行为

❌ **不要**:
- 只检查代码就说"ready"
- 只看首页就说"完成"
- 用无头浏览器就说"测试过了"
- 跳过任何验证阶段
- 隐瞒发现的问题
- "应该可以"、"应该没问题"

✅ **要**:
- 完整执行三个阶段
- 使用有头浏览器
- 测试所有功能
- 保留截图证据
- 发现问题立即修复
- 诚实报告状态

## 与其他 Skills 的关系

本 skill 整合了以下三个验证 skills：
- **ready-to-use-verification**: 开箱即用验证（浏览器功能测试）
- **verification-before-completion**: 完成前验证（代码级验证）
- **webapp-testing**: Web应用测试（自动化测试）

**执行顺序**: 代码级 → 浏览器功能 → 深度测试

## 最后的话

**在用户打开浏览器之前，你应该已经打开过至少 3 次。**

这不是浪费时间，这是对用户时间的尊重。

自我迭代系统不是"可选项"，而是"必选项"。
