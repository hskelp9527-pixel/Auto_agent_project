# 自我迭代系统 (Self-Evolution System)

## 核心使命

**在用户看到项目之前，发现问题并修复。**

不要让用户成为你的测试员。在说"完成了"、"ready"、"可以启动了"之前，必须完成自我迭代验证流程。

## 何时触发

**必须使用此系统的情况**：
- 完成任何 Web 应用开发后
- 修复重要 bug 并准备提交后
- 添加新功能并准备展示后
- 用户要求"启动项目"、"部署"、"可以用了"之前
- 任何即将呈现工作成果给用户的时刻

**触发关键词**：
- "完成了"
- "ready"
- "可以启动了"
- "给用户看"
- "准备部署"
- "项目做完了"

## 自我迭代验证流程

### 阶段 0: 设计系统质量检查 (design-system-check) ⭐ **新增**

**目标**: 确保项目具备电商级设计系统标准

**为什么要检查设计系统？**
- 功能完整 ≠ 设计完整
- 设计系统质量直接影响用户体验
- 电商级应用必须具备企业级设计系统
- 避免"功能强大但视觉效果差"的问题

**执行步骤**：

1. **核心依赖检查**
   ```bash
   # 检查是否具备shadcn/ui生态
   cat package.json | grep -E "class-variance-authority|tailwind-merge|@radix-ui"
   ```

   必备依赖：
   - ✅ class-variance-authority (CVA)
   - ✅ tailwind-merge (className优化)
   - ✅ tailwindcss-animate (动画)
   - ✅ @radix-ui/* (无障碍组件)
   - ✅ next-themes (暗色模式)

2. **Tailwind配置检查**
   ```bash
   # 检查是否有完整的设计令牌
   cat tailwind.config.ts | grep -E "darkMode|hsl\(var\(--"
   ```

   必备配置：
   - ✅ darkMode: ["class"]
   - ✅ HSL颜色变量 (hsl(var(--...)))
   - ✅ 动画插件 (tailwindcss-animate)

3. **CSS变量系统检查**
   ```bash
   # 检查globals.css中的设计令牌
   cat app/globals.css | grep -E "var\(--|:root|\.dark"
   ```

   最少要求：
   - ✅ 亮色主题: 20+个变量
   - ✅ 暗色主题: 20+个变量
   - ✅ @layer base配置

4. **工具函数检查**
   ```bash
   # 检查cn()函数
   cat lib/utils.ts | grep -E "twMerge|cn"
   ```

5. **组件质量检查**
   ```bash
   # 检查Button组件是否使用CVA
   cat components/ui/Button.tsx | grep -E "cva|VariantProps"
   ```

   标准要求：
   - ✅ 使用class-variance-authority
   - ✅ 使用VariantProps类型
   - ✅ 无硬编码颜色
   - ✅ 无className重复

6. **无障碍性检查**
   ```bash
   # 查找常见问题
   grep -r 'aria-label.*aria-label' app/ components/
   ```

   应该为0个结果（无重复ARIA属性）

7. **评分和报告**
   - 优秀 (90-100分): 继续下一阶段
   - 良好 (70-89分): 建议改进后继续
   - 需要改进 (50-69分): 必须修复关键问题
   - 不合格 (<50分): 停止，必须重构设计系统

**退出条件**：
- ✅ 核心依赖齐全
- ✅ CSS变量系统完整
- ✅ 组件使用CVA
- ✅ 评分 ≥ 70分

**时间估算**: 5-10分钟

### 阶段 1: 代码级验证 (verification-before-completion)

**目标**: 确保代码层面没有明显问题

**执行步骤**：
1. 检查代码语法和编译错误
2. 运行 linter 和类型检查
3. 运行所有测试（如果有）
4. 检查是否有 console.error、未定义的变量
5. 验证 imports 和 dependencies

**验证命令示例**：
```bash
# TypeScript/Next.js
npm run build
npm run lint
npx tsc --noEmit

# 检查常见错误
grep -r "console.error" app/
grep -r "TODO" app/
grep -r "FIXME" app/
```

**退出条件**：
- ✅ 构建成功，无错误
- ✅ Linter 通过，无严重警告
- ✅ 无明显的代码错误

### 阶段 2: Web应用功能验证 (ready-to-use-verification)

**目标**: 确保应用在浏览器中实际运行正常

**执行步骤**：
1. **启动开发服务器**
   ```bash
   npm run dev &
   # 等待服务器完全启动（2-5秒）
   ```

2. **使用有头浏览器打开**
   ```bash
   # 必须使用 --headed 参数
   agent-browser open http://localhost:PORT --headed
   ```

3. **验证核心功能**（按顺序执行）
   - [ ] 首页正常加载
   - [ ] 侧边栏/导航栏存在
   - [ ] 没有布局错误
   - [ ] 控制台无错误：`agent-browser errors`

4. **测试所有导航链接**
   ```bash
   # 对每个导航项：
   agent-browser click @nav-link
   agent-browser wait --load networkidle
   agent-browser snapshot -i
   agent-browser errors
   ```

5. **测试核心功能**
   - [ ] 新建/创建功能
   - [ ] 编辑功能
   - [ ] 删除功能
   - [ ] 搜索/筛选功能
   - [ ] 表单提交
   - [ ] 数据展示

6. **保留证据**
   ```bash
   agent-browser screenshot /path/to/evidence.png
   agent-browser get url
   ```

**退出条件**：
- ✅ 所有主要页面可访问
- ✅ 所有导航链接工作正常
- ✅ 核心功能正常工作
- ✅ 浏览器控制台无错误
- ✅ 有截图证据

### 阶段 3: 深度测试 (webapp-testing)

**目标**: 使用自动化测试工具进行全面验证

**执行步骤**：
1. **选择测试方法**：
   - 快速验证：使用 agent-browser 手动测试
   - 深度验证：编写 Playwright 自动化测试

2. **如果需要自动化测试**：
   ```bash
   # 创建测试脚本
   # 使用 webapp-testing skill 的模式
   python scripts/with_server.py --server "npm run dev" --port 3000 \
     -- python your_automation.py
   ```

3. **测试覆盖**：
   - [ ] 用户注册/登录流程（如果有）
   - [ ] CRUD 操作完整性
   - [ ] 表单验证
   - [ ] 错误处理
   - [ ] 边界情况
   - [ ] 性能检查（加载时间）

**退出条件**：
- ✅ 关键用户流程测试通过
- ✅ 无崩溃或严重bug
- ✅ 用户体验流畅

## 决策树

```
开发完成
    ↓
[阶段0] 设计系统质量检查 ⭐
    ├─ 评分 ≥ 70分 → 进入阶段1
    ├─ 评分 50-69分 → 修复关键问题 → 重新检查
    └─ 评分 < 50分 → 重构设计系统 → 重新检查
         ↓
[阶段1] 代码级验证
    ├─ 失败 → 修复问题 → 重新开始
    └─ 通过 → 进入阶段2
         ↓
    [阶段2] 浏览器功能验证
         ├─ 失败 → 修复问题 → 重新开始
         └─ 通过 → 进入阶段3
              ↓
         [阶段3] 深度测试
              ├─ 失败 → 修复问题 → 重新开始
              └─ 通过 → ✅ 可以呈现给用户
```

## 常见失败模式和修复

### 阶段0 常见问题（设计系统）
- **缺少shadcn/ui**: 无设计系统标准
  - 修复: `npx shadcn-ui@latest init`
- **无CVA**: variants管理混乱
  - 修复: `npm install class-variance-authority`
- **硬编码颜色**: 无法统一主题
  - 修复: 使用CSS变量 `hsl(var(--primary))`
- **className重复**: 样式冲突
  - 修复: 使用`tailwind-merge`的`cn()`函数
- **无暗色模式**: 用户体验受限
  - 修复: 安装`next-themes`并配置
- **无障碍性差**: ARIA属性错误
  - 修复: 使用Radix UI组件

### 阶段1 常见问题
- **构建失败**: 语法错误、类型错误、依赖问题
- **Linter 错误**: 代码规范问题、未使用变量
- **Import 错误**: 路径错误、模块未导出

### 阶段2 常见问题
- **服务器启动失败**: 端口占用、配置错误
- **页面空白**: 组件错误、数据未加载
- **导航不工作**: 路由配置错误、Link 组件问题
- **控制台错误**: JavaScript 运行时错误、API 调用失败
- **功能缺失**: TODO 未实现、空函数

### 阶段3 常见问题
- **用户体验差**: 加载慢、交互卡顿
- **边界情况**: 空数据、错误数据未处理
- **表单问题**: 验证不足、提交失败

## 时间估算

| 项目规模 | 阶段0 | 阶段1 | 阶段2 | 阶段3 | 总计 |
|---------|------|------|------|------|------|
| 小型 | 3-5分钟 | 2-3分钟 | 5-10分钟 | 5分钟 | 15-23分钟 |
| 中型 | 5-8分钟 | 5分钟 | 15-20分钟 | 10分钟 | 35-43分钟 |
| 大型 | 8-10分钟 | 10分钟 | 30-40分钟 | 20分钟 | 68-80分钟 |

**原则**: 宁可多花40分钟验证（含设计系统检查），不要让用户发现1个问题。

## 核心原则

1. **证据优先**: 不相信"应该可以"，只相信实际测试结果
2. **用户视角**: 从用户角度测试，不是开发者角度
3. **完整性**: 测试所有声称可用的功能
4. **诚实**: 发现问题如实报告，不隐瞒
5. **迭代**: 发现问题 → 修复 → 重新验证 → 呈现

## 禁止行为

❌ **不要**:
- 只检查代码就说"ready"
- 只看首页就说"完成"
- 用无头浏览器就说"测试过了"
- 跳过任何验证阶段
- 隐瞒发现的问题
- "应该可以"、"应该没问题"

✅ **要**:
- 完整执行三个阶段
- 使用有头浏览器
- 测试所有功能
- 保留截图证据
- 发现问题立即修复
- 诚实报告状态

## 与其他 Skills 的关系

本 skill 整合了以下三个验证 skills：
- **ready-to-use-verification**: 开箱即用验证（浏览器功能测试）
- **verification-before-completion**: 完成前验证（代码级验证）
- **webapp-testing**: Web应用测试（自动化测试）

**执行顺序**: 代码级 → 浏览器功能 → 深度测试

## 最后的话

**在用户打开浏览器之前，你应该已经打开过至少 3 次。**

这不是浪费时间，这是对用户时间的尊重。

自我迭代系统不是"可选项"，而是"必选项"。
