# 开箱即用验证 (Ready-to-Use Verification)

## 核心原则

**"开箱即用"意味着用户打开就能用，没有任何问题。**

在声明任何 Web 应用"完成"、"ready"、"可以用了"之前，**必须**完成以下验证流程。

## 铁律

```
NO "READY" CLAIMS WITHOUT HEADFUL BROWSER VERIFICATION
```

没有通过有头浏览器完整验证 = 不能声称"ready"或"完成"

## 必须执行的验证步骤

### 1. 启动开发服务器
```bash
# 必须在后台启动
npm run dev  # 或相应的启动命令
# 等待服务器完全启动（通常需要 2-5 秒）
```

### 2. 使用有头浏览器打开
```bash
# 必须使用 --headed 参数（可以看到浏览器窗口）
agent-browser open http://localhost:PORT --headed
# 或使用其他有头浏览器工具
```

**关键**: 必须能看到浏览器窗口，不是为了截图，而是为了实际看到页面渲染效果。

### 3. 验证核心功能（按顺序）

#### A. 首页验证
- [ ] 页面正常加载
- [ ] 侧边栏导航存在（如果应该有）
- [ ] 顶部导航栏存在（如果应该有）
- [ ] 没有明显的布局错误
- [ ] 检查控制台：`agent-browser errors`（必须无错误）

#### B. 导航测试（测试所有主要链接）
- [ ] 点击每个导航链接
- [ ] 等待页面加载：`agent-browser wait --load networkidle`
- [ ] 验证页面标题正确
- [ ] 验证页面内容正确
- [ ] 验证没有控制台错误

**必须测试的导航项**（根据实际应用调整）：
- 首页
- 订单管理
- 产品管理
- 库存管理
- 客户管理
- 财务管理
- 系统设置
- 等等...

#### C. 核心功能测试
- [ ] 测试"新建"功能（如果有）
- [ ] 测试"搜索"功能（如果有）
- [ ] 测试"编辑"功能（如果有）
- [ ] 测试"删除"功能（如果有）
- [ ] 验证表单提交工作正常
- [ ] 验证数据展示正确

#### D. 响应式验证（可选但推荐）
- [ ] 测试不同视口大小
- [ ] 验证移动端显示（如果需要）

### 4. 最终检查清单

在声明"ready"之前，必须确认：

- [ ] 服务器正常启动，无启动错误
- [ ] 所有主要页面可访问
- [ ] 所有导航链接工作正常
- [ ] 核心功能（CRUD操作）正常工作
- [ ] 浏览器控制台无错误
- [ ] 没有明显的布局问题
- [ ] 表单可以正常提交
- [ ] 数据正确显示

### 5. 记录验证结果

```bash
# 保留截图作为证据
agent-browser screenshot /path/to/screenshot.png

# 记录测试的页面
agent-browser get url
```

## 常见错误模式

### ❌ 错误做法
```bash
# 只检查服务器启动
npm run dev  # 看到启动成功就说"ready"
```

### ❌ 错误做法
```bash
# 只用代码静态分析
# 看起来代码没问题就说"ready"
```

### ❌ 错误做法
```bash
# 只用无头浏览器
agent-browser open http://localhost:3000  # 默认无头
# 看不到实际渲染效果
```

### ✅ 正确做法
```bash
# 1. 启动服务器
npm run dev &

# 2. 等待启动完成
sleep 5

# 3. 使用有头浏览器
agent-browser open http://localhost:3000 --headed

# 4. 验证首页
agent-browser snapshot -i
agent-browser errors

# 5. 测试导航
agent-browser click @nav-orders
agent-browser wait --load networkidle
agent-browser snapshot -i
agent-browser errors

# 6. 测试核心功能
agent-browser click @btn-create
agent-browser wait --load networkidle
agent-browser fill @input-name "测试订单"
agent-browser click @btn-submit
agent-browser wait --load networkidle
agent-browser errors

# 7. 检查所有页面
# （重复上述步骤）

# 8. 只有在所有测试通过后才说"ready"
```

## 特定场景检查清单

### Next.js 应用
- [ ] 侧边栏/导航栏正常显示
- [ ] 页面路由工作正常
- [ ] Link 组件导航流畅
- [ ] 没有 hydration 错误
- [ ] 客户端状态管理正常

### React 应用
- [ ] 组件正常渲染
- [ ] State 更新正常工作
- [ ] 事件处理器正常触发
- [ ] 没有 React 警告

### 数据驱动应用
- [ ] 数据加载正常
- [ ] 表格/列表正确显示
- [ ] 搜索/筛选功能工作
- [ ] 分页功能工作（如果有）

### 表单应用
- [ ] 表单验证工作
- [ ] 提交成功/失败提示正确
- [ ] 表单重置功能工作
- [ ] 必填字段验证正确

## 验证失败的常见原因

1. **导航不工作** - 路由配置错误
2. **侧边栏缺失** - Layout 组件未正确使用
3. **控制台错误** - JavaScript 错误、未定义的变量
4. **功能不完整** - TODO 注释、空实现
5. **样式问题** - CSS 未加载、布局错乱
6. **数据问题** - API 调用失败、mock 数据错误

## 何时使用此 Skill

**必须使用的情况**：
- 完成 Web 应用开发后
- 修复重要 bug 后
- 添加新功能后
- 重构代码后
- 用户要求"启动项目"或"部署"之前
- 任何涉及"ready"、"完成"、"可以用了"的声明之前

## 为什么这很重要

从实际经验总结：
- 用户反馈"经常会发生这种事，启动之后报错"
- 用户明确要求"你应该先打开一次有头模式操作浏览器，对功能进行验证"
- 用户强调"我希望是开箱即用，而不是现在这样各种问题"

**"开箱即用"是基本要求，不是额外要求。**

```
## 验证清单

- [ ] 有头浏览器打开
- [ ] 首页加载正常
- [ ] 所有导航链接测试通过
- [ ] 核心功能测试通过
- [ ] 控制台无错误
- [ ] 截图已保存
- [ ] 可以声明 "ready"
```

## 最后的话

**宁可多花 5 分钟验证，不要让用户发现问题。**

验证不是"可以跳过"的步骤，而是"必须执行"的流程。

在用户打开浏览器之前，你应该已经打开过至少 3 次。
